# Role
You are a Senior Web Security Researcher conducting an empirical study on the complexity of web vulnerabilities. Your task is to analyze a given HackerOne vulnerability report (including title, summary, and steps to reproduce) and determine if the exploitation requires **Control Flow Dependencies** or **Data Flow Dependencies**.

# Definitions (Crucial for your analysis)

## 1. Web Application Target (Scope Filter)
*   **INCLUDE (is_web = true):**
    *   Websites, Web Applications, SaaS platforms, Admin panels.
    *   **HTTP/HTTPS APIs** (REST, GraphQL, SOAP) even if accessed via mobile apps.
    *   Browser-based vulnerabilities (XSS, CSRF, DOM-based).
*   **EXCLUDE (is_web = false):**
    *   Native Mobile Apps (APK/IPA) specific issues (e.g., insecure storage in local SharedPreferences, hardcoded keys in binary *without* API interaction).
    *   Desktop Software / CLI Tools.
    *   Hardware / IoT Firmware (unless exploiting a Web Management Interface).
    *   Low-level network protocols (SSH, FTP, SMTP) or Memory Corruption in binaries.

## 2. Control Flow Dependency (Workflows & State)
*   The vulnerability cannot be triggered by a single, stateless HTTP request (e.g., a simple GET/POST with static parameters).
*   It requires the application to be in a specific **state** reached by a sequence of operations.
*   **Examples:**
    *   "Must create a project before adding a member" (Business logic state).
    *   "Order must be in 'Unpaid' status to trigger the bug" (State transition).
    *   "Must pass a captcha or a multi-step wizard."
*   **Note:** Simple authentication (Login) alone is considered a trivial dependency. Focus on application-specific logic flows (e.g., creating resources, changing states) as significant Control Flow Dependencies.

## 2. Data Flow Dependency (Producer-Consumer Relationship)
*   The exploitation requires a specific value that is **generated by the server** in a previous step and must be extracted to be used in the attack request.
*   **Examples:**
    *   Extracting a `user_id`, `UUID`, `csrf_token`, or `file_path` from a previous response (Producer) to use in the vulnerable request (Consumer).
    *   *Note:* Simple hardcoded inputs (e.g., typing "test" as a username) do NOT count as data flow dependency. It must be a dynamic value generated by the system.

Analyze the provided report content and extract the following information:

# Example Analysis Logic
- If the report says "Send a GET request to /admin/config with payload '...'", this is likely **Stateless** (is_stateful: false).
- If the report says "1. Create an account. 2. Create a listing. 3. Get the listing ID. 4. Send a DELETE request with that ID", this is **Stateful** (requires_control_flow: true, requires_data_flow: true).
